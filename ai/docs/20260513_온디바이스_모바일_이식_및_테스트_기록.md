# 2026-05-13 온디바이스 모바일 이식 및 테스트 기록

## 1. 작업 배경

이날의 목표는 PC 환경에서 검증이 끝난 V6 24단어 TCN TFLite 모델을 모바일 앱으로 이식하고,
실시간 카메라 입력에서도 단어 인식과 문장 조합이 동작하도록 전체 파이프라인을 연결하는 것이었다.

당시 기준으로는 다음이 이미 정리되어 있었다.

- PC `inference_realtime_tflite_v6.py` 에서는 단일 단어 인식이 가능했다.
- V6 TFLite 모델 파일, 라벨 맵, 설정 파일은 준비되어 있었다.
- 모바일 앱에는 기존 수어 인식용 카메라 파이프라인과 MediaPipe landmark 추출 구조가 있었다.

문제는, “모델을 모바일에 올리는 것”과 “PC와 같은 입력 분포로 실시간 추론시키는 것”은 전혀 다른 문제라는 점이었다.
그래서 2026-05-13에는 우선 모바일 쪽에 전체 경로를 이식하고, 바로 테스트 가능한 상태를 만드는 데 초점을 맞췄다.

---

## 2. 초기 전략

당시 세운 기본 전략은 다음과 같았다.

1. 모바일 assets 에 V6 TFLite 모델과 label map 을 배치한다.
2. 모바일 추론 경로가 30 x 332 feature 를 만들 수 있게 encoder 와 contract 를 맞춘다.
3. 실시간 카메라 프레임에서 MediaPipe landmark 를 추출한다.
4. 일정 길이 이상의 프레임이 모이면 빠르게 추론을 시작한다.
5. 단어 후보를 smoothing 하고, 손이 사라진 뒤 문장을 마무리한다.
6. 눈썹 위치 차이를 이용해 평서문/의문문도 같이 분류한다.

즉, Python 쪽 로직을 그대로 복제하기보다, 모바일에 맞는 구조로 옮기되
입력 차원과 후처리 전략은 최대한 PC와 비슷하게 가져가는 방향이었다.

---

## 3. 모바일 이식 작업

## 3-1. 모델/계약 정리

먼저 모바일 앱이 V6 TFLite 모델을 읽을 수 있도록 관련 파일과 계약을 정리했다.

적용 항목:

- `best_sign_model_v6_float16.tflite` 를 모바일 assets 쪽에 배치
- `label_map_v6.json` 반영
- 클래스 수를 23개 기준으로 맞춤
- feature dimension 을 332로 맞춤
- confidence threshold 도 모바일 쪽 계약에 반영

관련 정리 대상:

- `SignModelVariant.kt`
- `SignModelOutput.kt`
- `SignModelContract.kt`

이 단계에서는 “모바일이 적어도 같은 모델을 읽는다”는 상태를 만드는 것이 우선이었다.

---

## 3-2. 332차원 feature encoder 연결

그 다음으로 landmark 기반 feature encoder 를 모바일 쪽에 붙였다.

당시 구현한 방향:

- 왼손 63
- 오른손 63
- 얼굴 90
- 포즈 21
- 손-얼굴 거리 95

위 조합으로 총 332차원 feature 를 생성하도록 구성했다.

핵심 의도는 PC Python 코드에서 `extract_keypoints()` 가 하던 역할을 모바일 `LandmarkFeatureEncoder.kt` 로 옮기는 것이었다.

이 단계에서는 손 landmark 가 안 잡히는 순간을 어떻게 처리할지, 이전 손 좌표를 유지할지, 0으로 채울지 같은 세부 정책도 같이 들어가기 시작했다.

---

## 3-3. 빠른 추론 시작과 smoothing 적용

실시간성 때문에 30프레임이 모두 모일 때까지 기다리지 않고,
15프레임 정도만 모여도 추론을 시작하는 fast start 전략을 도입했다.

적용한 생각은 다음과 같았다.

- 사용자는 수어를 1초 이상 멈춰서 보여주지 않기 때문에 초기 응답 속도가 중요하다.
- 15프레임 이후부터는 마지막 프레임 복제나 upsampling 으로 30프레임 입력 형태를 만든다.
- 같은 단어가 몇 프레임 이상 반복되면 안정된 예측으로 인정한다.

이를 위해:

- `SignSequenceBuffer.kt`
- `SignPredictionStabilizer.kt`

같은 구조가 들어갔다.

당시 가정은 “초반에는 약간 흔들려도, smoothing 하면 최종 단어는 맞을 것”이었다.

---

## 3-4. 손 소실 처리와 문장 종료 로직

연속 문장을 위해서는 단어 인식 자체보다 “어디서 시작하고 어디서 끝나는가”가 중요했다.

그래서 다음 로직을 함께 넣었다.

- 손이 잠깐 안 보여도 바로 초기화하지 않는다.
- 일정 시간 이상 손이 모두 사라지면 문장을 종료한다.
- 그 전까지 누적된 단어들을 최종 문장으로 출력한다.

이때의 아이디어는:

- 손이 한두 프레임 놓치는 것은 흔한 일이라 grace period 가 필요하다.
- 너무 빨리 끊으면 단어가 여러 개로 분절된다.
- 너무 늦게 끊으면 다음 단어나 다음 문장이 앞 문장을 오염시킨다.

따라서 no-hand grace period 와 sentence buffer 가 함께 들어갔다.

---

## 3-5. 눈썹 기반 문장 타입 분류

PC 쪽에서 사용하던 눈썹-눈 거리 기반 평서문/의문문 판별도 모바일에 같이 옮기려고 했다.

접근 방식:

- 중립 표정의 baseline 을 몇 프레임 동안 모은다.
- 현재 눈썹 위치가 baseline 대비 얼마나 올라갔는지 비교한다.
- threshold 이상이면 의문문으로 간주한다.

이 단계에서의 목표는 단어 인식만이 아니라,
최종 문장을 “평서문” 또는 “의문문”으로 함께 마무리하는 것이었다.

---

## 4. 실제 테스트에서 드러난 문제들

이식 자체는 끝났지만, 바로 테스트해보니 예상보다 훨씬 많은 문제가 나왔다.
2026-05-13의 핵심은 사실 “이식 완료”보다 “문제가 무엇인지 드러난 날”에 가까웠다.

## 4-1. 모바일 입력과 PC 입력이 정말 같은가에 대한 의심

처음부터 가장 크게 걸린 부분은,
모바일 카메라 입력이 PC 웹캠 입력과 같은 feature 분포를 만들고 있는지 확신할 수 없다는 점이었다.

특히 의심한 지점은 다음과 같았다.

- 모바일은 세로 화면, PC는 가로 화면 기반이라는 점
- 카메라 프리뷰와 실제 분석 입력의 비율이 다를 수 있다는 점
- MediaPipe 결과 좌표를 모바일 쪽에서 따로 스케일링하고 있다는 점
- 프런트 카메라 미러링 여부가 PC와 다를 수 있다는 점

이 시점에는 아직 “근본 원인”까지는 특정하지 못했지만,
입력 분포 차이가 핵심일 가능성이 높다고 느끼기 시작했다.

---

## 4-2. aspect ratio 왜곡에 대한 가설

당시 가장 강하게 의심했던 것 중 하나는 비율 왜곡이었다.

모바일 구현에서는 세로 화면에 가로형 분석 프레임을 맞추는 과정이 있었고,
encoder 내부에서도 일부 좌표에 별도 스케일 보정이 들어가 있었다.

그래서 이런 가설을 세웠다.

- 모바일이 16:9 입력을 기대하는 모델에 9:16에서 왜곡된 값을 넣고 있을 수 있다.
- 손/얼굴/포즈 landmark 좌표가 스케일 보정 때문에 원래보다 부풀려졌을 수 있다.
- 이 차이가 모델 입장에서는 전혀 다른 동작처럼 보일 수 있다.

이 가설은 이후 2026-05-14에 dump 비교로 실제로 검증되지만,
2026-05-13 시점에는 “매우 유력한 의심점” 수준이었다.

---

## 4-3. 카메라 해상도와 FPS 차이에 대한 가설

또 다른 의심은 모바일 카메라 설정 자체였다.

당시 비교 포인트:

- 모바일: 낮은 해상도 또는 제한된 분석 해상도
- 모바일: 상대적으로 낮은 FPS
- PC: 16:9 웹캠, 30FPS 기반 실시간 추론

세운 아이디어:

- 분석 해상도를 1920x1080, 30FPS 에 최대한 가깝게 맞춰보자.
- TCN 모델은 프레임 간 변화량에도 민감할 수 있으므로 FPS가 중요하다.
- 프레임이 성기면 연결 동작이 과도하게 뭉개질 수 있다.

이때는 “단순히 모델 문제”가 아니라, 카메라 파이프라인 자체가 입력 품질에 큰 영향을 준다고 봤다.

---

## 4-4. 손이 잠깐 안 보이는 구간 처리 문제

손이 화면 밖으로 나가거나 MediaPipe 가 손을 놓치는 구간을 어떻게 처리하느냐도 문제였다.

초기 모바일 로직에서는:

- 손이 안 보일 때 0 좌표를 넣기도 하고
- 어떤 경우에는 이전 프레임 손 좌표를 채워넣는 방식도 고려했다.

이 과정에서 생긴 걱정은 다음과 같았다.

- 0 좌표를 너무 많이 넣으면 모델이 이상한 패턴을 배울 수 있다.
- 이전 손 좌표를 너무 오래 유지하면 이미 끝난 단어가 계속 남는다.
- 단어 사이 transition 을 오히려 더 왜곡할 수 있다.

즉, “손 소실 처리”는 단순한 보정 기능이 아니라
문장 분리 성능까지 영향을 주는 핵심 요소로 보였다.

---

## 4-5. 미러링 문제

프런트 카메라의 미러링 여부도 강하게 의심했다.

PC 쪽은 사용자 입장에서는 좌우 반전이 자연스러워 보이지만,
실제 모델 입력이 어떤 방향으로 들어가는지가 중요했다.

모바일에서는:

- Preview 에서 보이는 화면
- 분석용 Bitmap
- MediaPipe 에 전달되는 입력

이 세 경로가 서로 다를 수 있었다.

그래서 한동안은 “사용자가 보는 방향”보다
“모델이 실제로 어떤 방향의 landmark 를 받는가”를 우선적으로 맞추는 쪽으로 생각이 이동했다.

---

## 4-6. confidence 해석 문제

초기에는 TFLite 출력 해석도 다시 봐야 한다고 생각했다.

문제의식:

- PC 쪽에서 threshold 를 어떻게 적용했는지
- 모바일이 logits 를 확률처럼 쓰고 있는지
- unknown rejection 기준이 맞는지

이 단계에서는 “출력 해석이 달라서 생기는 문제일 수도 있다”는 가설도 같이 세웠다.
그래서 threshold 와 accepted/rejected 판정 로직도 함께 만졌다.

---

## 5. 2026-05-13 시점 결론

이날 작업을 통해 얻은 가장 중요한 결론은 다음과 같았다.

1. 모바일에 모델을 올리는 것 자체는 가능했다.
2. 하지만 모바일 입력이 PC와 동일하다고 가정하면 안 된다.
3. 문제는 단일 지점이 아니라 전처리, 해상도, 미러링, 손 소실 처리, 후처리가 모두 얽혀 있었다.
4. 따라서 다음 단계에서는 추측성 수정만으로는 한계가 있다.

즉, 2026-05-13은 “모바일 이식을 완료한 날”이기도 하지만,
동시에 “입력 분포를 직접 비교할 방법이 필요하다는 결론에 도달한 날”이었다.

이 결론은 다음 날 작업 방향에 직접 영향을 줬다.

- 모바일 dump 기능 추가
- PC TFLite dump 기능 추가
- mobile dump vs pc dump 직접 비교

이 세 가지가 2026-05-14의 핵심 전략으로 이어졌다.

## 6. 다음 날로 이어진 과제

2026-05-13 작업 종료 시점에 남아 있던 과제는 아래와 같았다.

1. 모바일이 실제로 모델에 넣는 `30 x 332` 값을 저장할 수 있어야 한다.
2. PC TFLite 추론 결과와 같은 형식으로 비교할 수 있어야 한다.
3. 단일 단어와 연속 문장 모두에서 어떤 단계에서 틀어지는지 수치로 확인해야 한다.
4. 이후에는 감으로 수정하지 말고, dump 비교 기반으로 원인을 특정해야 한다.

결국 이 날의 기록은 “해결 완료 보고서”라기보다,
문제를 구성하는 요소들을 드러내고 다음 날의 디버깅 전략을 만드는 중간 정리라고 볼 수 있다.

2026-05-13에는 V6 TFLite 기반 모바일 온디바이스 수어 인식 경로를 앱에 실제로 이식했지만,
실전 테스트를 통해 **문제의 본질이 모델 자체보다 모바일 입력 파이프라인과 PC 입력 분포의 불일치에 있을 가능성**을 강하게 확인했고,
그 결과 다음 날부터 dump 비교 중심 디버깅으로 방향을 전환하게 되었다.

# 2026-05-11 온디바이스 연속 수어 적용 전략 초기 계획

## 요약

기존 7개 단어 Transformer 모델은 PC의 가로 영상 기반 테스트에서는 안정적으로 동작했지만, 모바일 실기기 환경에서는 인식 안정성이 낮았다. 원인은 모델 자체만이 아니라 모바일 입력 파이프라인, 세로 화면, 실시간 30프레임 버퍼링, 손 검출 누락, 좌우 반전 가능성, 학습 데이터 구성 차이까지 함께 확인해야 한다고 판단했다.

따라서 2026-05-11 작업은 바로 모델 구조를 바꾸기보다, 온디바이스 적용을 위한 실험 작업 공간을 만들고, 학습 단어와 데이터 수집 정책을 다시 정리하며, 모바일 추론 실패 원인을 분리해서 확인할 수 있는 기준을 세우는 데 초점을 두었다.

## 문제 인식

기존 모델은 isolated word classification 구조다.

```text
입력: [1, 30, 332]
출력: [1, 7]
```

이 구조는 단어 하나가 잘린 입력을 분류하는 데는 사용할 수 있지만, 실제 앱에서는 사용자가 준비 동작, 단어 동작, 전환 동작, 정지 동작을 연속으로 수행한다. 모바일에서 최근 30프레임을 그대로 모델에 넣으면, 학습 데이터처럼 "단어 동작 전체가 30프레임으로 압축된 입력"이 아니라 동작의 일부 또는 전환 구간이 들어갈 수 있다.

핵심 가설은 다음과 같다.

```text
PC 학습/테스트: 단어 영상 전체를 30프레임으로 리샘플링
모바일 실시간: 현재 시점 기준 최근 30프레임을 sliding window로 입력
```

따라서 모바일 추론이 실패할 경우 다음 항목을 순서대로 확인해야 한다.

```text
1. 332차원 feature 순서가 학습 코드와 동일한가
2. 30프레임 sequence가 학습 데이터와 같은 의미를 갖는가
3. 손 검출 프레임 비율이 충분한가
4. 세로 화면, 카메라 mirror, 좌우 손 매핑 차이가 있는가
5. TFLite 출력 confidence와 margin이 threshold를 통과하는가
```

## 유지할 기준

초기 실험에서는 모델 구조와 feature 추출 방식을 크게 바꾸지 않는다.

```text
모델: Transformer
feature: 332차원 landmark feature
sequence length: 30
정규화: shoulder center / shoulder width 기반 정규화
추론 형식: TFLite 온디바이스 추론
```

PC와 모바일의 좌표 추출 방식 차이는 계속 의심해야 한다. 특히 PC는 `mediapipe.solutions.holistic`, 모바일은 `holistic_landmarker.task`를 사용하므로, 같은 영상에서 두 방식의 30x332 결과를 비교하는 검증이 필요하다.

## 데이터 생성 정책

숫자가 붙은 단어 후보는 모두 하나로 합치지 않는다. 예를 들어 `가다`, `가다3`, `가다4`가 있을 때 실제 동작이 다르면 같은 라벨로 섞는 것이 성능을 떨어뜨릴 수 있다.

정책은 다음과 같이 확정했다.

```text
1. 목표 단어와 숫자 variant를 후보로 수집한다.
2. 후보를 WORD ID 기준으로 묶는다.
3. 각 목표 단어마다 데이터 개수가 가장 많은 WORD ID 하나만 대표로 선택한다.
4. 선택된 WORD ID의 mp4만 base 단어 폴더로 저장한다.
5. 선택되지 않은 WORD ID는 같은 라벨에 섞지 않는다.
```

예시는 다음과 같다.

```text
후보: 가다 30개, 가다3 20개, 가다4 80개
선택: 데이터가 가장 많은 후보 1개
저장: processed_npy_v6/가다/*.npy
제외: 선택되지 않은 다른 가다 variant
```

## v100 데이터 생성 방식

초기에는 AIHub shell로 ZIP을 내려받으면서 `.npy`를 생성하는 방식과 S3 우선 처리 방식을 검토했다. 하지만 다운로드 시간이 오래 걸려, 팀원이 v100에 이미 받아둔 AIHub WORD ZIP 파일을 직접 사용하는 방식으로 전환했다.

로컬 ZIP 기반 파이프라인은 다음 스크립트로 정리했다.

```text
docs/Ondevice_v1/local_zip_to_npy_pipeline.py
```

처리 흐름은 다음과 같다.

```text
1. 여러 --zip-root 아래의 *_real_word_video.zip을 스캔한다.
2. AI_Hub_Video_Mapping_v6.json에 있는 mp4만 찾는다.
3. 이미 processed_npy_v6/{단어}/{파일명}.npy가 있으면 스킵한다.
4. 없는 mp4만 임시 추출한다.
5. MediaPipe로 332차원 npy를 생성한다.
6. 임시 mp4는 삭제한다.
```

## 모바일 추론 점검 방향

모바일 테스트에서 인식이 안 될 경우, 바로 모델만 문제로 보지 않고 다음 로그 기준으로 분리한다.

```text
Sequence ready가 안 나옴
→ 30프레임 버퍼가 안 차거나 손 프레임 조건에서 막힌 상태

Sequence ready는 나오지만 Inference result가 안 나옴
→ TFLite 입력 또는 실행 문제

Inference result는 나오지만 Prediction emitted가 안 나옴
→ confidenceThreshold 또는 marginThreshold가 너무 높은 상태

Inference result가 나오지만 gloss가 계속 틀림
→ 입력 구간, 좌표 정규화, 좌우 반전, 세로 화면 차이 가능성
```

확인할 로그 필터는 다음과 같다.

```text
SignPipeline
```

중요 로그는 다음과 같다.

```text
Sequence ready. frames=30, handFrames=..., sequenceSize=9960
Inference result. sequenceSize=9960, gloss=..., confidence=..., margin=...
Prediction emitted. gloss=..., confidence=...
```

## 모바일 수정 방향

모바일 코드는 우선 관찰 가능한 상태로 만드는 것을 1차 목표로 했다.

1차 수정 방향은 다음과 같다.

```text
1. sequence ready 시점 로그 추가
2. 추론 결과 confidence와 margin 로그 추가
3. 모델 asset을 transformer_mobile 산출물로 교체
4. 앱이 새 TFLite 모델과 label_map.json을 바라보도록 경로 수정
```

추가로 검토할 개선 방향은 다음과 같다.

```text
1. 최근 30프레임 방식 대신 동작 구간 segment를 모은 뒤 30프레임으로 리샘플링
2. 좌우 손 mirror 여부 확인
3. 세로 preview와 모델 입력 좌표계 분리
4. threshold와 margin 기준 완화 또는 debug 조정
5. Python TFLite와 Android TFLite의 동일 입력 top-k 비교
```

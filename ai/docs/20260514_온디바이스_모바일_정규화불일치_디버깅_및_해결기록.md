# 2026-05-14 온디바이스 모바일 정규화불일치 디버깅 및 해결기록

## 1. 작업 배경

2026-05-13까지의 작업으로 V6 TFLite 기반 모바일 온디바이스 인식 경로는 연결되어 있었지만,
실제 테스트 결과는 PC와 큰 차이가 있었다.

당시 관측된 대표 증상은 다음과 같았다.

- 어떤 단어를 해도 `가다`가 1.000으로 반복되던 시기
- 그 상태를 넘긴 뒤에는 단일 단어를 수행해도 전혀 다른 단어가 top-1으로 나오던 시기
- 연속 문장에서는 앞 단어와 뒤 단어가 섞이거나, no-hand 구간 이후에도 이전 단어가 다시 튀는 현상

중요한 점은, PC `inference_realtime_tflite_v6.py` 에서는 같은 모델이 잘 동작하고 있었다는 것이다.
즉 문제를 “모델 자체”로 보기보다,
**모바일이 모델에 넣는 입력이 PC와 얼마나 다른가**를 확인하는 쪽으로 방향을 잡아야 했다.

이날의 목표는 명확했다.

1. 모바일과 PC의 `30 x 332 sequenceFrames` 를 직접 비교할 수 있게 만든다.
2. 감으로 후처리를 만지는 대신, 실제 입력 차이를 보고 원인을 특정한다.
3. 원인이 특정되면 그 차이를 최소한의 수정으로 해결한다.

## 5. 단일 단어 테스트로 문제 범위 축소

연속 문장으로 보면 segmentation, transition, stabilizer 문제가 섞여 보이기 때문에,
먼저 단일 단어 위주로 문제를 분리해서 보기로 했다.

대표 테스트 단어:

- `기차`
- `놀다`
- `무섭다`

초기 결과는 좋지 않았다.

예시 패턴:

- `기차`를 했는데 `돕다`, `놀다`, `싫다`가 뜨는 경우
- `놀다`를 했는데 `걷다`, `싫다`가 뜨는 경우

문제가 stabilizer가 아니라 top-1 자체라는 점이었다.

즉:

- “맞는 단어가 나오는데 확정이 안 된다”가 아니라
- “애초에 모델 입력이 달라서 다른 단어로 분류된다”가 더 맞았다.

## 7. 핵심 원인 특정

원인을 추적한 결과, 가장 결정적인 차이는 `normalize_mode` 였다.

### PC 기준

`ai/models/tcn_mobile/train_config_v6.json`

- `stats.normalize_mode = "none"`

즉 PC TFLite 파이프라인은 **RAW 입력**을 기대하고 있었다.

### 모바일 기준

`LandmarkFeatureEncoder.kt`

- 기본 normalize mode 가 `SHOULDER`

그리고 실시간 인식 엔진은 별도 지정 없이 이 encoder 기본값을 사용하고 있었다.

즉 실제 상황은 아래와 같았다.

- PC: `RAW`
- Mobile realtime: `SHOULDER`

이 정규화 불일치 때문에 landmark block 뿐 아니라
distance block 까지 완전히 다른 분포로 들어가고 있었다.

이 시점에서 문제의 핵심은 거의 확정됐다.

**모바일이 잘못된 좌표계로 모델을 호출하고 있었다.**

## 8. 해결 아이디어와 실제 수정

이제 해야 할 일은 명확했다.

- 실시간 인식 엔진에서 사용하는 `LandmarkFeatureEncoder` 를 `RAW` 모드로 바꾼다.

적용 파일:

- `mobile/app/src/main/java/com/ssafy/mobile/core/vision/RealSignRecognitionEngine.kt`

변경 방향:

- 기존: `LandmarkFeatureEncoder()` 또는 `SHOULDER` 기본값 사용
- 변경: `LandmarkFeatureEncoder(normalizeMode = LandmarkFeatureNormalizeMode.RAW)`

이 수정은 단순하지만 매우 중요했다.

왜냐하면:

- 모델을 바꾸는 것이 아니라
- 모바일이 모델에 넣는 입력 좌표계를 PC와 일치시키는 작업이었기 때문이다.

## 12. 프리뷰 UX 문제와 실제 모드 보정

단일 단어 인식이 안정화된 뒤에는,
사용자가 상반신을 화면에 충분히 넣기 어렵다는 UX 문제가 드러났다.

### 디버그 모드

- 세로 화면에서 프리뷰가 비교적 넓게 보였음

### 실제 모드

- 카메라 영역이 가로 비율에 가까워 상반신을 다 넣기 어려웠음
- 너무 뒤로 가야 해서 사용성이 떨어졌음

이를 해결하기 위해 두 단계로 프리뷰를 손봤다.

1. 디버그 모드 프리뷰는 `FIT_CENTER` 에서 `FILL_CENTER` 로 바꿔 검은 여백을 줄였다.
2. 실제 모드 대화 화면에서는 프리뷰 영역 비율을 `16:9` 에서 `9:16` 쪽으로 맞춰 세로 박스로 더 크게 보이게 바꿨다.

중요한 점은,
이 변경은 **표시용 프리뷰 UX 수정**이지 분석 입력 파이프라인 자체를 바꾼 것은 아니라는 점이다.

즉:

- 인식 로직은 유지
- 사용자가 상반신과 손을 프레임에 넣기 더 쉬워짐

실제 모드에서도 동작 확인이 가능해졌다.

## 13. 중간에 다시 흔들렸던 현상 해석

정규화 수정 후에도 일부 로그에서는 첫 15프레임 창이 다른 단어로 잠깐 튀는 경우가 있었다.

예:

- `놀다` 테스트 초반에 첫 추론 창은 `감기`
- 다음 창부터 `놀다`로 수렴
- 최종 `Detected` 와 dump 는 `놀다`

이 현상은 정규화가 다시 틀어진 것이 아니라,
초기 15프레임 창에 동작 진입 구간이 섞이면서 생긴 일로 해석했다.

중요한 판단 기준은 다음과 같았다.

- 초기 창이 잠깐 흔들리는 것은 허용 가능
- 최종 `Detected` 와 dump gloss 가 맞으면 정상 범주
- 연속 문장까지 들어가서 이 흔들림이 실제 문장 조합을 망치는지가 다음 확인 포인트

즉 이 단계에서는 “완벽히 한 프레임도 안 흔들림”보다
“최종 단어가 올바르게 안정화되는가”를 더 중요하게 봤다.

## 14. 오늘 결론

2026-05-14에 얻은 결론은 매우 명확하다.

### 결론 1. 핵심 원인은 정규화 불일치였다.

- PC TFLite V6는 `normalize_mode = none`
- 모바일 실시간 엔진은 `SHOULDER` 입력 사용
- 이 mismatch 때문에 모바일 입력 분포가 모델 기대값과 크게 달라졌다.

### 결론 2. `RAW` 로 맞추자 인식이 정상화됐다.

아래 세 단어가 모두 안정적으로 맞았다.

- `기차`
- `놀다`
- `무섭다`

### 결론 3. dump 비교 전략이 실제로 문제를 해결했다.

이번 이슈는 단순 로그만 보면

- stabilizer 문제
- segmentation 문제
- 모델 문제
- 카메라 문제

처럼 보일 수 있었지만,
실제로는 `sequenceFrames` 비교를 통해 입력 좌표계 차이를 수치로 확인하면서 해결할 수 있었다.

즉 이후 비슷한 문제가 생겨도,
가장 먼저 해야 할 일은 **mobile dump vs pc dump 비교**이다.

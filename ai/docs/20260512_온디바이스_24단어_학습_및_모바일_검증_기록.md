### Ondevice_v1 학습 코드 정리

`docs/Ondevice_v1` 폴더가 정리된 상태를 확인했다. 현재 남아 있는 핵심 파일은 학습, 평가, TFLite 변환, PC 실시간/영상 추론에 필요한 파일 위주다.

24개 단어 `processed_npy_v6` 데이터를 바로 사용할 수 있도록 다음 코드를 수정했다.

```text
train.py
dataset.py
convert_to_tflite.py
evaluate_holdout.py
inference_realtime_transformer.py
inference_realtime_tflite.py
inference_video_tflite.py
```

수정 방향은 다음과 같다.

```text
1. 기본 모델 출력 폴더를 v6_24words_transformer_baseline으로 변경
2. train.py의 환경변수 기준을 V6_*로 정리하고 기존 V5_*도 fallback으로 유지
3. 학습 대상 class를 target_words_v6.py의 24개 단어로 고정
4. 변환/추론 스크립트가 best_sign_model_v6_*.pt, label_map_v6_*.json을 우선 탐색하도록 수정
5. PC TFLite/Transformer 실시간 추론도 V6 모델 폴더를 기본값으로 사용하도록 수정
```

수정 후 `docs/Ondevice_v1`의 모든 Python 파일에 대해 문법 검사를 수행했고 통과했다.

`train.py` 상단에 v100 학습 실행 예시 명령을 docstring으로 추가했다. 코드 실행에는 영향을 주지 않고, 서버에서 바로 참고할 수 있는 용도다.

학습 하이퍼파라미터도 추가로 정리했다. 기존에는 `d_model`, `num_heads`, `num_layers`, `weight_decay`, scheduler factor/patience가 코드 중간에 하드코딩되어 있었으므로, 상단 설정값과 `V6_*` 환경변수로 조정 가능하게 변경했다. 해당 값들은 `train_config_v6_*.json`에도 저장되도록 했다.

### 시도 1 학습 설정

현재 코드를 24개 단어 모델 학습의 첫 번째 시도로 정의한다.

시도 1의 목표는 단순 오버피팅이 아니라 PC와 모바일 모두에서 적용 가능한 일반화 성능을 얻는 것이다. 따라서 기존 v5처럼 epoch마다 검증 지표를 자세히 출력하되, best model 저장 기준은 train acc가 아니라 validation macro F1 중심으로 변경했다.

시도 1의 주요 설정은 다음과 같다.

```text
데이터: processed_npy_v6
클래스 수: 24
샘플 수: TOTAL 1910
입력 shape: 30 x 332
모델: Transformer encoder
d_model: 128
num_heads: 8
num_layers: 3
batch_size: 16
learning_rate: 0.001
weight_decay: 0.0
epoch: 최대 120
early stopping: 사용
early stopping patience: 25
best model 기준: validation macro F1
balanced sampler: 기본 비활성화
class weights: 기본 비활성화
```

데이터가 거의 균일하게 맞춰졌기 때문에 balanced sampler는 기본값을 끈다. `병원`, `엄마`만 75개이고 나머지는 80개라서 sampler나 class weight가 꼭 필요하지 않다고 판단했다.

증강은 완전히 끄지 않고 약하게 적용한다.

```text
target_samples_per_class: 120
noise_std: 0.005
time_scale_range: 0.95 ~ 1.05
time_shift_range: -2 ~ 2
```

증강을 너무 강하게 주면 모바일 좌표 입력과 분포가 달라질 수 있으므로, 시도 1에서는 약한 시간축 변화와 작은 노이즈만 사용한다.

매 epoch마다 유지할 검증 출력은 다음과 같다.

```text
train loss
train acc
val loss
val acc
macro F1
train-val acc gap
label histogram
prediction histogram
confusion matrix
class recall
learning rate
```

현재 실행 명령은 다음과 같다.

```bash
V6_DATA_DIR=./processed_npy_v6 \
V6_MODEL_DIR_NAME=v6_24words_transformer_baseline \
V6_EPOCHS=120 \
V6_BATCH_SIZE=16 \
V6_LR=0.001 \
V6_WEIGHT_DECAY=0.0 \
V6_BALANCED_SAMPLER=0 \
V6_USE_AUGMENTATION=1 \
V6_TARGET_SAMPLES_PER_CLASS=120 \
V6_EARLY_STOPPING=1 \
V6_EARLY_STOPPING_PATIENCE=25 \
python train.py
```

이 설정으로 학습을 실행한 뒤 결과를 이어서 기록한다.

### Holdout 데이터 분리 방식

`evaluate_holdout.py`를 의미 있게 사용하려면 holdout 데이터가 학습에 사용된 데이터와 겹치면 안 된다. 따라서 `processed_npy_v6` 전체로 바로 학습한 뒤 일부 파일을 holdout으로 복사하는 방식은 사용하지 않는다.

`prepare_holdout_split.py`를 추가해 학습 전에 다음 두 폴더로 분리하도록 했다.

```text
train_npy_v6
holdout_test_v6
```

기본 설정은 각 클래스에서 최대 10개를 holdout으로 분리한다. 80개 클래스는 `train=70, holdout=10`, 75개 클래스는 `train=65, holdout=10`이 된다.

실행 명령은 다음과 같다.

```bash
python prepare_holdout_split.py \
  --source ./processed_npy_v6 \
  --train-output ./train_npy_v6 \
  --holdout-output ./holdout_test_v6 \
  --holdout-count 10 \
  --overwrite
```

이후 학습은 `processed_npy_v6`가 아니라 `train_npy_v6`를 사용한다.

```bash
V6_DATA_DIR=./train_npy_v6 \
V6_MODEL_DIR_NAME=v6_24words_transformer_baseline \
V6_EPOCHS=120 \
V6_BATCH_SIZE=16 \
V6_LR=0.001 \
V6_WEIGHT_DECAY=0.0 \
V6_BALANCED_SAMPLER=0 \
V6_USE_AUGMENTATION=1 \
V6_TARGET_SAMPLES_PER_CLASS=120 \
V6_EARLY_STOPPING=1 \
V6_EARLY_STOPPING_PATIENCE=25 \
python train.py
```

학습 후 holdout 검증은 다음처럼 실행한다.

```bash
V6_HOLDOUT_DIR=./holdout_test_v6 \
V6_MODEL_DIR=v6_24words_transformer_baseline \
python evaluate_holdout.py
```

### 학습 저장/GPU/병렬 처리 확인

`train.py`는 CUDA가 사용 가능하면 `cuda`를 사용하고, 모델과 배치 데이터를 GPU로 이동해서 학습한다. best 모델은 검증 macro F1이 개선될 때마다 같은 run 파일(`best_sign_model_v6_{번호}.pt`)에 덮어쓴다. 새 학습 run은 기존 파일을 덮지 않도록 suffix 번호를 증가시킨다.

데이터 로딩/증강 병렬 처리를 위해 `V6_NUM_WORKERS`, `V6_PIN_MEMORY`, `V6_PERSISTENT_WORKERS` 옵션을 추가했다. v100에서는 우선 `V6_NUM_WORKERS=2`, `V6_PIN_MEMORY=1`로 시작하고, GPU 사용률이 낮거나 CPU/RAM 여유가 충분하면 `V6_NUM_WORKERS=4`까지 올려볼 수 있다.

### 시도 2 실행 전 에러

v100에서 `train.py`만 최신이고 `dataset.py`가 이전 버전인 상태로 실행되어 `create_dataloaders_v5() got an unexpected keyword argument 'num_workers'` 에러가 발생했다. 해결 방법은 `dataset.py`도 함께 업로드하거나, 최신 `train.py`에서 구버전 `dataset.py`를 감지해 worker 옵션 없이 실행하도록 처리하는 것이다. 정상적인 병렬 로딩을 위해서는 `dataset.py`도 최신 버전으로 맞추는 것이 좋다.

또한 v100 서버 기준으로 `models/v6_24words_transformer_1/` 폴더가 없으면 자동 생성 로직상 `_1`부터 만들어진다. 시도 1 결과가 `v6_24words_transformer_baseline`에 남아 있다면 이를 `v6_24words_transformer_1`로 정리한 뒤 다시 실행해야 다음 결과가 `_2`에 저장된다.

`models/v6_24words_transformer_2/` 학습 결과도 낮게 나왔다.

랜덤 수준보다는 조금 높지만, 여전히 학습이 정상적으로 진행됐다고 보기 어렵다. v5 코드의 이름/호환성 잔재가 혼선을 만들 수 있으므로 실제 학습 경로를 v6 전용 이름으로 분리했다.

- `KSLTransformerV6` 사용

### MLP 시도 결과

`V6_MODEL_TYPE=mlp`, `V6_NORMALIZE_MODE=none`, augmentation off 조건에서 성능이 크게 개선됐다.

```text
Best F1: 0.8405
Val Acc@BestF1: 84.13%
Best Train Acc: 92.51%
model_dir=models/v6_24words_mlp_1
```

데이터는 정상이고, Transformer 구조/학습 방식이 이번 v6 332차원 입력에 잘 맞지 않았던 것으로 판단한다. MLP는 30프레임 x 332차원을 펼쳐서 전체 패턴을 분류하는 단어 분류기이며, 단어 단위 클립에서는 강하지만 연속 수어의 연결 동작을 자동으로 무시하는 모델은 아니다. 연속 수어에서는 sliding window, confidence threshold, margin, smoothing, lock/unlock 후처리가 필요하다.

MLP 테스트를 위해 변환/추론 코드가 `train_config_v6.json`의 `model_type`과 `normalize_mode`를 읽도록 수정했다. 특히 MLP는 `normalize_mode=none`으로 학습했으므로 실시간 추론에서도 shoulder normalization을 적용하지 않아야 한다.

기존 파일명 `inference_realtime_transformer.py`는 transformer 전용처럼 보이지만, 내부는 `model_type`을 읽어 MLP도 실행할 수 있도록 수정했다. 혼동을 줄이기 위해 같은 진입점으로 동작하는 `inference_realtime_v6.py`를 추가했다.

영상 파일 테스트도 현재 기준 모델인 `.pt` MLP로 해야 하므로 `inference_video_v6.py`를 추가했다. 기존 `inference_video_tflite.py`는 TFLite 변환 이후 검증용으로만 사용한다.

## 🚀 시도 4: 연속 동작 포착을 위한 파이프라인 전면 개편 (TCN + None 클래스)

### 1. 개요
* **발생 원인:** 이전 시도(MLP/Transformer)에서 정규화(Normalization) 부재로 인한 도메인 시프트 발생. 또한 단어와 단어 사이의 의미 없는 동작(휴식, 손 내리기 등)을 포착하지 못하여 지속적으로 오작동(False Positive) 발생.
* **대응 목표:**
  1. **TCN(Temporal Convolutional Network) 적용:** 연속된 프레임에서 국소적인 시계열 특징을 잘 잡아내는 TCN을 기본 모델로 채택.
  2. **Shoulder Normalization 필수화:** 사용자 체형/거리와 무관하게 일관된 좌표계 획득.
  3. **None 클래스 추가:** 수어가 아닌 일상 동작(안경 올리기, 대기 상태 등) 데이터를 `none` 클래스로 직접 수집하여 오탐지 확률 억제. (학습 단어 배열에서 성능을 저하시키던 '먹' 단어를 제외하고 24단어 유지)
  4. **실시간 추론 파이프라인 고도화:** `inference_realtime_v6.py`에 Sliding Window(큐), Confidence Threshold(>0.8), Temporal Smoothing(연속 N프레임 동일 예측 시 출력) 로직 구현.

### 2. 환경 세팅 및 데이터 준비
* **단어 구성 변경:** `target_words_v6.py`에서 '먹' 제외, 'none' 추가.
* **데이터 업로드:** 팀원이 직접 촬영한 none 데이터를 V100 서버의 `processed_npy_v6/none/` 에 적재.
* **학습 파라미터 변경:**
  * `V6_MODEL_TYPE=tcn`
  * `V6_NORMALIZE_MODE=shoulder`
  
### 3. V100 서버 학습 명령어
```bash
# V100 서버 환경에서 학습 실행
V6_DATA_DIR=./processed_npy_v6 V6_MODEL_DIR_NAME=v6_24words_tcn_1 V6_MODEL_TYPE=tcn V6_NORMALIZE_MODE=shoulder python train.py
```

### 4. 다음 계획
학습이 끝난 `best_sign_model_v6.pt` 파일을 로컬 모델 폴더로 가져와, `inference_realtime_v6.py`를 실행하여 연속 동작 사이사이의 'none' 판단과 단어 Thresholding이 정상적으로 작동하는지 확인.

## 🚀 시도 5: 실시간 추론 트러블슈팅 및 '친구' 단어 블랙홀 현상 해결

### 1. 현상 파악 (단어 고립 추론 테스트)
V100에서 학습된 TCN 모델(`best_sign_model_v6.pt`)을 로컬 캠에 연동하여 단일 동작별 실시간 테스트를 진행한 결과, 나머지 대부분의 단어가 매우 잘 인식되었으나 일부 단어에서 특정 단어로 예측이 쏠리는 **오탐지(False Positive)** 현상이 발견되었습니다.

**[오탐지 발생 목록]**
* 아기 -> **친구**
* 무섭다 -> **친구**
* 기차 -> **친구**
* 밥 -> 감사
* 목마르다 -> 장난감
* 감기 -> 감기, 기차 (오락가락함)
* 엄마 -> 손을 높이 들어야 인식됨. 낮게 하면 '감기'로 인식됨.
* 장난감 -> 기차, 놀다

### 2. 원인 분석
* **'친구' 단어의 블랙홀 현상:** 수어 '친구' 동작은 두 손을 맞대고 있는 형태인데, 이 동작의 특징(랜드마크 거리 등)이 다른 여러 단어(아기, 무섭다, 기차 등 손을 모으거나 교차하는 동작)와 너무 유사하여 모델이 조금만 애매해도 모두 '친구'로 묶어버리는 블랙홀 현상이 발생했습니다.
* **높이(Y축)에 따른 인식률 저하 ('엄마' vs '감기'):** Shoulder 정규화를 통해 거리는 보정되었으나, '엄마'(턱/볼 부근)와 '감기'(코/얼굴 부근)가 모두 얼굴 근처에서 이루어지다 보니 사용자가 손을 올리는 미세한 높이 차이에 따라 모델이 혼동을 겪고 있습니다.

### 3. 해결 방안 및 조치 (실행)
1. **블랙홀 단어 '친구' 학습 제외:** 다른 단어의 인식률까지 크게 갉아먹고 있는 주범인 '친구'를 학습 데이터에서 완전히 배제하여, '아기/무섭다/기차' 등이 원래의 정답 공간을 찾아갈 수 있도록 유도합니다. (`target_words_v6.py`에서 삭제 완료, 총 23단어로 재편)
2. **모델 재학습:** '친구'가 제거된 상태(23 클래스)로 V100 서버에서 다시 TCN 모델을 학습합니다.

### 4. 재학습 결과 (친구 제외, 23클래스)
* **Val Acc:** 99.45% (이전 99.21% 대비 상승)
* **F1 Score:** 0.9945 (이전 0.9917 대비 상승)
* **결과 분석:**
  * 가설이 완벽하게 적중했습니다! '친구'에게 억울하게 오탐지 당하던 **'아기', '무섭다', '기차'의 Recall이 모두 1.0000(100%)으로 완벽하게 복구**되었습니다.
  * 새로 도입했던 **'none' 클래스 역시 12/12 (100%) 정확도**를 기록하며 수어가 아닌 동작을 완벽히 방어해 내는 기염을 토했습니다.
  * 극소수의 혼동(엄마->웃다 1건, 감사->손 1건) 외에는 Confusion Matrix가 완벽한 대각선을 그렸습니다.

### 5. 로컬 PC 실시간 연속 동작 검증
* 위에서 검증된 `v6_24words_tcn_2` 모델을 로컬로 가져와 `inference_realtime_v6.py`를 통해 실제 사용 환경(PC 웹캠)에서 테스트를 진행합니다.

### 6. 실시간 검증 결과 및 한계점 분석
23클래스 모델을 PC 웹캠 환경에서 테스트한 결과, '친구' 블랙홀 현상은 완벽하게 해결되었으나 몇 가지 새로운 엣지 케이스(Edge Case)가 발견되었습니다.

**[추가 발견된 오탐지 / 미탐지]**
1. **싫다 -> none (미탐지):** '싫다' 동작이 너무 짧고 간결해서, 모델이 80% 이상의 확신을 갖지 못하거나 스무딩(5프레임) 필터에 걸러져 `none`으로 무시되는 현상.
2. **감기 -> 아프다:** 두 수어 모두 손을 얼굴(코, 이마 등)에 가져다 대는 동작이라, 2D 웹캠 기반의 미디어파이프(Z축 깊이 인식의 한계) 환경에서 서로의 특징이 강하게 겹침.
3. **병원 -> 기차 / 감기 -> 기차:** 손이 교차하거나 위아래로 움직이는 특정 프레임의 순간적인 모습이 '기차'의 궤적과 유사하게 잡힘.

**[해결을 위한 임계값(Threshold) 튜닝]**
'싫다'와 같이 짧은 동작이 `none`으로 묻히는 현상(False Negative)을 해결하기 위해 `inference_realtime_v6.py`의 민감도를 높였습니다.
- `CONFIDENCE_THRESHOLD`: 0.8 -> **0.7** (더 낮은 확신도로도 정답 인정)
- `PREDICTION_SMOOTHING`: 5 -> **3** (더 짧은 순간의 찰나의 동작도 포착)
- `STABLE_MIN_COUNT`: 3 -> **2** (2프레임만 유지되어도 단어 확정)

**[연속 동작 및 손 겹침 시 끊김 현상 해결]**
* **원인:** '배고프다'처럼 손을 휙 뒤집거나, '감기'처럼 한 손에서 양손으로 바뀔 때 MediaPipe가 찰나의 순간(1~3프레임) 손을 인식하지 못하고 놓치는 현상이 발생함. 기존 코드에서는 손이 단 1프레임만 안 보여도 30프레임 메모리 큐(Queue)를 강제로 초기화해버려서 인식이 뚝뚝 끊김.
* **해결 조치:** 손 유실 유예(Grace Period) 타이머 적용. 손이 안 보일 때 즉시 초기화하지 않고 빈 좌표(0)를 큐에 넣으며 흐름을 유지하다가, **15프레임(0.5초) 연속으로 손이 안 보일 때만 큐를 초기화**하도록 `inference_realtime_v6.py` 로직 수정. 이를 통해 손이 교차하거나 뒤집히는 연속 동작도 매우 부드럽게 인식됨.

### 7. 단일 고립 단어(Isolated Word) 실시간 검증 최종 성공
위에서 적용한 3가지 해결책(1. '친구' 제외, 2. 민감도 튜닝, 3. 프레임 유실 유예)을 모두 적용한 결과, 드디어 로컬 PC 웹캠 환경에서 **학습된 모든 단어(23클래스)가 끊김 없이, 정확하게, 개별적으로 완벽히 인식되는 것을 확인**하였습니다. 

이제 단일 단어 인식의 안정성이 확보되었으므로, 이를 바탕으로 실제 사용 환경과 동일한 **연속 수어 문장 인식(Continuous Sign Language Recognition) 테스트**로 넘어갈 준비가 완료되었습니다.

### 8. 연속 문장 검증 및 필터링 고도화
PC 웹캠 환경에서 단어를 끊지 않고 연속으로 이어서 동작하는 테스트를 진행했습니다. (예: 아기 -> 밥 -> 주다)

**[문제점 발견: 과도기(Transition) 프레임 오탐지]**
테스트 결과 `아기 감기 아프다 밥 아프다 주다 기차` 와 같이, 의도한 단어들 사이에 '감기', '아프다', '기차' 같은 노이즈 단어들이 대거 끼어드는 현상이 발생했습니다.
* **원인:** 단어와 단어 사이로 손이 이동하는 찰나의 순간에, 손이 얼굴을 스치거나(감기, 아프다) 손이 교차하며(기차) 발생하는 순간적인 모양을 모델이 포착해버림. 이전에 '싫다'를 잡기 위해 민감도를 너무 높여둔(Threshold 0.7, Smoothing 3) 부작용이 나타남.

**[해결 조치: 빡빡한 필터링 적용]**
찰나의 노이즈 동작을 걸러내기 위해 `inference_realtime_v6.py`의 기준을 대폭 강화했습니다.
- `CONFIDENCE_THRESHOLD`: 0.7 -> **0.85** (정말 확실할 때만 인정)
- `PREDICTION_SMOOTHING`: 3 -> **7** (순간적인 스침이 아니라 확실하게 동작을 7프레임 이상 머물러야 인정)
- `STABLE_MIN_COUNT`: 2 -> **5** (최소 5프레임 이상 연속으로 같은 판단을 내려야만 확정)

**[문제점 재발견: 짧은 단어 미탐지 (False Negative)]**
* 임계값을 0.85, 스무딩을 7프레임으로 너무 높였더니 과도기 노이즈는 사라졌으나, 사용자가 예상했던 대로 '싫다'와 같이 물리적으로 동작이 매우 짧은 단어들이 스무딩 허들(0.23초)을 넘지 못하고 통째로 무시되는 현상이 다시 발생했습니다. (예: "기차 무섭다 싫다" -> "기차 무섭다" 로 출력됨)

**[최종 해결 조치: 황금 밸런스(타협점) 도출]**
과도기 노이즈 필터링과 짧은 단어 인식이라는 딜레마 사이에서 최적의 중간값을 찾기 위해 다시 세팅을 튜닝했습니다.
- `CONFIDENCE_THRESHOLD`: 0.85 -> **0.80** 
- `PREDICTION_SMOOTHING`: 7 -> **5** (최소 5프레임 이상 유지)
- `STABLE_MIN_COUNT`: 5 -> **4**
이 세팅을 통해 짧은 동작도 살리면서, 의미 없는 노이즈 스침은 걸러내도록 타협점을 맞추었습니다.

### 9. 연속 문장 검증 한계 및 궁극적 해결 방안 도출
가장 밸런스가 좋은 0.80 / 5프레임 세팅으로 "아기 밥 주다" 연속 동작을 테스트한 결과, 여전히 과도기 구간에서 `아기 아프다 밥 아프다 주다` 처럼 노이즈 단어가 끼어드는 현상이 확인되었습니다.

**[핵심 원인 분석: 과도기(Transition) 데이터의 부재]**
이 현상은 코드(소프트웨어)의 임계값 문제가 아니라, **AI 모델이 학습한 데이터 자체의 근본적인 한계**임이 밝혀졌습니다.
* 현재 `none` 클래스 학습 데이터는 '가만히 서 있기', '안경 올리기' 등 정적인 동작 위주로 수집되었습니다.
* 하지만 실제 수어 간의 연속 동작('아기' 위치에서 '밥' 위치로 손이 휙 이동하는 궤적)은 모델 입장에서 한 번도 본 적 없는 미지의 영역입니다.
* 모델은 자신이 아는 23개 단어 중 가장 비슷한 것을 억지로 끼워 맞추려다 보니, 손이 얼굴을 스치고 지나가는 찰나를 '아프다'라고 착각(False Positive)하게 되는 것입니다.

**[궁극적인 해결 방안 (CSLR의 정석)]**
연속 수어 인식(Continuous Sign Language Recognition)을 완벽하게 구현하기 위해서는, **'의미 없이 손이 이동하는 과도기 궤적' 자체를 `none` 클래스에 포함시켜 학습**시켜야 합니다.
* **추후 과제:** 사용자가 손을 위아래로 휘젓거나, 특정 수어 위치에서 다른 위치로 의미 없이 빠르게 이동하는 동작을 20~30개 정도 추가 촬영하여 `none` 데이터에 병합 후 재학습하면 이 과도기 노이즈 문제가 완벽하게 사라질 것입니다.

### 10. 문장 누적 출력 및 비수지기호(의문문) 로직 추가
단순히 단일 단어를 실시간으로 띄워주던 방식에서 벗어나, 실제 서비스의 문장 번역 파이프라인에 맞추어 추론 코드(`inference_realtime_v6.py`)를 최종 고도화했습니다.

**[추가된 기능 1: 문장 누적 및 종료(Flush) 처리]**
* 손이 화면에 등장해서 수어를 진행하는 동안 포착되는 단어들을 `sentence_buffer`에 중복 없이 차곡차곡 누적합니다.
* 수어가 끝난 뒤 **손이 화면 밖으로 사라지면(15프레임 이상 유실 시)** 누적해 둔 단어들을 하나로 합쳐 `최종 문장: 단어 단어 단어` 형태로 터미널에 한 번에 출력하고 버퍼를 비웁니다.

**[추가된 기능 2: 비수지기호(눈썹) 기반 의문문 판별 로직 이식]**
* 예전 V5 모델에 사용했던 '눈썹 치켜올림' 판별 로직을 V6에 완벽하게 통합했습니다.
* **평상시:** 손이 화면에 없을 때의 평소 눈썹 간격(Neutral Gap)의 중앙값을 지속적으로 기록합니다.
* **수어 중:** 손이 등장해서 수어를 하는 도중, 단 한 프레임이라도 눈썹 간격이 평소보다 `0.055` 이상 크게 벌어지면 해당 문장을 '의문문' 상태로 임시 저장합니다.
* **최종 출력:** 손이 내려가고 최종 문장이 출력될 때 꼬리표를 달아줍니다. (예: `기차 무섭다 (의문문)`, `아기 밥 주다 (평서문)`)

### 11. 최종 결론 및 모바일 환경 테스트 계획
지금까지 진행한 수많은 튜닝(블랙홀 단어 '친구' 제외, Threshold 0.8 / Smoothing 5프레임 최적화, 손 유실 Grace Period 적용 등)을 거친 현재의 **23클래스 TCN 모델이 역대 가장 높은 안정성과 인식률을 보여주는 베스트 모델(Best Model)**로 확인되었습니다.

**[PC 로컬 테스트 총평]**
* 단일 단어 인식은 예외 없이 완벽하게 동작함을 확인했습니다.
* 문장 추출을 위한 '연속적인 수어 동작(CSLR)' 역시, 과도기 노이즈의 근본적인 한계(데이터 부족)를 소프트웨어 로직으로 최대한 억제하여 어느 정도 매끄럽게 문장 단위로 잘 인식되고 출력되는 것을 검증 완료했습니다.

**[Next Step: 모바일 검증]**
이처럼 PC 웹캠 환경에서 충분한 검증을 마쳤고 실전 활용 가능성이 확인되었으므로, **이 최종 모델(`v6_24words_tcn_2`의 pt 파일)을 안드로이드/모바일 환경으로 변환 및 탑재하여 실제 스마트폰 앱 환경에서도 동일하게 부드러운 연속 추론이 가능한지 모바일 테스트를 진행할 예정**입니다.

### 12. 모바일 탑재를 위한 TFLite 변환 및 Softmax 로직 검증
PC에서 검증된 `v6_24words_tcn_2` 모델을 안드로이드 환경에 올리기 위해 TFLite 포맷으로 변환을 진행합니다. (명령어: `python convert_to_tflite.py --model-dir v6_24words_tcn_2`)

**[TFLite 채택 사유]**
안드로이드 네이티브(Java/Kotlin) 환경에서 MediaPipe의 랜드마크 출력값 배열을 직접 받아 실시간 딥러닝 추론을 돌리기에는, 호환성과 속도 측면에서 TensorFlow Lite(TFLite)가 업계 표준이자 가장 안정적인 선택지입니다.

**[Softmax에 대한 고찰 및 적용]**
모바일 변환 시 모델의 최종 출력단에 Softmax를 씌우는 것이 적합한가에 대한 딜레마가 제기되었습니다.
* **우려:** Softmax의 특성상 총합이 1.0이 되므로, 확신이 없는 동작에서도 특정 단어 하나가 억지로 높은 확률을 배정받아 오탐지를 일으킬 수 있지 않은가?
* **결론:** 모바일에도 **Softmax를 씌운 채로 내보내는 것이 정답**입니다.
  1. 모델이 확신이 없을 경우 Softmax는 확률을 23개 클래스에 잘게 쪼개어 분산시키므로, 어떤 단어도 저희가 설정해 둔 `0.80`이라는 빡빡한 임계값을 넘지 못해 자연스럽게 `none`으로 필터링됩니다.
  2. 무엇보다 저희가 직접 추가해 둔 `none` 클래스가 애매한 동작들의 확률을 홀로 0.99 이상 쓸어 담아가는 '강력한 방패' 역할을 해주기 때문에, 오탐지 확률이 억지로 치솟을 염려가 없습니다.
따라서 안드로이드 개발자가 모바일 코드 단에서 직관적으로 "0.8 이상일 때 출력"이라는 로직을 쉽게 짤 수 있도록, TFLite 모델 자체에 Softmax를 부착(Baking)하여 추출했습니다.


### 13. 모바일 이식 전 TFLite 로컬 PC 사전 검증 및 환경 세팅
모바일 파트원에게 모델을 넘기기 전, TFLite 모델이 PyTorch 환경과 100% 동일한 성능을 내는지 검증하기 위해 파이썬용 TFLite 추론 스크립트(`inference_realtime_tflite_v6.py`)를 제작하여 로컬 테스트를 진행했습니다.

**[TFLite 실시간 검증 스크립트 작성]**
* PyTorch 코드를 완전히 걷어내고 `tf.lite.Interpreter`를 사용하여 안드로이드 모바일 기기 내부와 완벽하게 동일한 추론 환경을 모사했습니다.
* TFLite 환경에서도 기존 파이토치 버전과 완벽하게 동일한 전/후처리(Threshold, Smoothing, Grace Period, 눈썹 의문문 로직 등)가 정상 작동하는 것을 확인했습니다.
* 터미널뿐만 아니라 **웹캠 화면(OpenCV 창) 좌측 상단에도 현재까지 누적된 문장과 의문문 판별 상태를 실시간으로 띄워주도록 코드를 고도화**하여 시각적인 피드백을 강화했습니다.

**[로컬 테스트 환경 트러블슈팅 (JAX & ml_dtypes 충돌)]**
* TFLite 검증을 위해 로컬 가상환경(venv)에 TensorFlow 2.15.0을 설치하던 중, 기존에 깔려있던 `jax` 라이브러리가 요구하는 `ml_dtypes` 버전과 TensorFlow가 설치한 구버전 `ml_dtypes` 간에 치명적인 버전 충돌이 발생했습니다.
* **해결 조치:** 현재 실시간 추론 파이프라인에서는 `jax`가 불필요하므로, 텐서플로우가 에러 없이 구동될 수 있도록 로컬 가상환경에서 `jax`와 `jaxlib`를 완전히 삭제(Uninstall)하여 깔끔하게 문제를 해결했습니다.

**[안드로이드(모바일) 이식 전략 도출]**
테스트 과정에서 안드로이드의 최신 Tasks API(`holistic_landmarker.task`)와 파이썬의 레거시 API(`mp.solutions.holistic`) 간의 차이에 대한 논의가 진행되었습니다.
* 현재 파이썬 환경에서는 구글 정책상 `holistic_landmarker.task`를 사용한 HolisticLandmarker API를 직접 지원하지 않습니다.
* 하지만 파이썬 내장 모델과 안드로이드의 `.task` 모델은 껍데기만 다를 뿐 **수학적으로 100% 동일한 모델**이므로 뿜어내는 좌표값과 순서가 완전히 같습니다.
* **결론 (안드로이드 이식 지침):** 모바일 파트원에게 "안드로이드 Tasks API로 뽑은 랜드마크 뭉치를 파이썬의 `extract_keypoints()` 함수와 **토씨 하나 틀리지 않고 동일하게 332개의 특정 좌표만 추출하고 어깨 기준 정규화(Shoulder Normalization)를 거친 뒤** TFLite 모델에 넣기만 하면 파이썬과 100% 동일한 결과가 나온다"는 명확한 이식 전략을 확립했습니다.

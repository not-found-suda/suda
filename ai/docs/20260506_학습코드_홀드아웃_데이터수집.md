# 2026-05-06 학습 코드 점검, 단어 축소, Holdout 평가, 데이터 수집 전략

## 작업 배경

10개 단어 학습 결과가 일정 수준까지 올라갔지만, 실제 실시간 테스트에서는 특정 단어가 계속 다른 단어로 잘못 인식되는 문제가 나타났다. 특히 `오다`, `자다`, `아프다`가 불안정했고, `가다`와 동작이 비슷한 단어들이 서로 섞이는 문제가 반복되었다.

5월 5일에는 별도 작업을 진행하지 않은 것으로 보고, 학습 코드 구성 및 초기 실험에 대한 정리는 이 날짜 작업 흐름에 통합한다.

## 학습 코드 구성 점검

기존 구현을 기반으로 현재 학습 코드 세트를 구성하고 점검했다.

```text
dataset.py
model.py
train.py
inference_realtime_transformer.py
convert_to_tflite.py
```

초기 학습 코드는 다음 흐름으로 동작하도록 정리했다.

```text
1. processed_npy에서 단어별 npy 파일 수집
2. 대표 단어 label map 생성
3. train/validation split
4. 30프레임 입력으로 padding 또는 sampling
5. Transformer 모델 학습
6. best F1 기준 모델 저장
7. label_map.json, train_config.json 함께 저장
```

모델 저장은 기존 파일을 덮어쓰지 않고 번호를 붙이는 방식으로 정리했다.

```text
best_sign_model_1.pt
label_map_1.json
train_config_1.json

best_sign_model_2.pt
label_map_2.json
train_config_2.json
```

또한 모바일 변환을 고려해 TFLite 변환 스크립트를 준비했고, softmax 누락을 막기 위해 변환 시 확률 출력을 유지하는 방향을 잡았다.

## 주요 관찰

10개 단어 학습에서 validation 수치는 괜찮게 보였지만, 실제 웹캠 추론에서는 다음과 같은 문제가 있었다.

```text
오다 -> 가다
자다 -> 가다
가다 -> 자다
아프다 -> 가다
```

이는 validation split이 원본 AI Hub 데이터 안에서 무작위로 나뉘었기 때문에, 실제 사용자 또는 모바일 촬영 환경에서의 일반화를 충분히 검증하지 못한다는 판단으로 이어졌다.

## 단어 수 조정

제한된 핵심 단어 범위에서 실제 시연 환경의 인식 안정성을 확보하기 위해, 오인식이 반복되는 단어를 단계적으로 제외하는 방향을 검토했다.

처음에는 `오다`, `아프다`를 제외한 8개 단어 실험을 진행했다. 이후 `자다`도 `가다`와 계속 혼동되면서 제한된 시연 범위의 안정성을 낮춘다고 판단해 제외했다.

최종적으로 한 시점에서는 아래 7개 단어를 핵심 단어 후보로 삼고, 실제 촬영 holdout 데이터 기준으로 안정성을 검증했다.

```text
기차, 장난감, 놀다, 가다, 일어나다, 병원, 조심
```

## Holdout 평가 도입

학습/검증 split만으로는 실제 사용자 촬영 데이터 성능을 알기 어렵기 때문에 `holdout_test`를 만들고, 학습에 사용하지 않은 직접 촬영 데이터를 평가하는 스크립트를 만들었다.

```text
evaluate_holdout.py
```

이 스크립트는 다음을 출력하도록 했다.

```text
모델 경로
label map 경로
holdout 디렉터리
평가 파일 수
누락된 클래스
클래스별 정확도
confusion matrix
오답 파일과 top3 확률
```

처음 holdout에는 `가다`, `일어나다`, `조심` 정도만 있었기 때문에 전체 검증이 아니라 부분 검증이었다. 이후 7개 단어 전체에 대해 직접 촬영한 holdout 데이터를 추가하는 방향을 잡았다.

## 데이터 수집 코드

기존 수집 스크립트를 기반으로 현재 feature 구조에 맞춘 수집 코드를 만들었다.

수집 구조는 다음과 같이 잡았다.

```text
1세트 = 3F + 1L + 1R
FRONT, FRONT, FRONT, LEFT, RIGHT
```

저장 구조는 다음과 같이 유지했다.

```text
processed_npy/{대표단어}/{변형단어}/{파일명}.npy
```

예시는 다음과 같다.

```text
processed_npy/가다/가다4/가다_가다4_ksw_001_FRONT.npy
processed_npy/일어나다/일어나다2/일어나다_일어나다2_ksw_001_FRONT.npy
```

## 얼굴 및 눈썹 landmark 판단

평서문과 의문문 구분을 위해 눈썹 데이터가 필요한지 논의했다. 현재 feature에는 얼굴 landmark 일부가 이미 포함되어 있고, 눈썹 주변 landmark도 포함되어 있음을 확인했다.

안경이 눈썹처럼 보이는지에 대한 우려도 있었지만, MediaPipe가 안경 자체를 인식한다기보다 얼굴 landmark를 추정하는 것이므로, 안경이 있어도 landmark가 완전히 다른 의미가 되는 것은 아니라고 판단했다.

## 촬영 방식 정리

촬영 시에는 카운트다운이 끝나는 시점부터 동작을 시작하고, 끝날 때는 마지막 자세를 잠시 유지하는 방식이 자연스럽다고 정리했다.

60fps 촬영도 가능하지만, 모델은 최종적으로 30프레임 입력으로 맞춰 학습하기 때문에 수집 코드에서 길이를 보정할 수 있다고 보았다.

## 학습 데이터와 holdout 분리 원칙

직접 촬영한 holdout 데이터는 학습에 섞지 않는 것이 원칙으로 정했다. 다만 시연 환경에서의 안정성 확인이 필요한 경우에는 새로 촬영한 별도 데이터만 학습 보강에 사용하고, 기존 holdout 파일은 검증 전용으로 유지하는 방식을 권장했다.

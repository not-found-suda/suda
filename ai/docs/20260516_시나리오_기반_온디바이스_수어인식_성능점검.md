# 2026-05-16 시나리오 기반 온디바이스 수어 인식 성능 점검

## 작업 목적
- 모바일 온디바이스 수어 인식 파이프라인의 실제 사용성을 점검한다.
- 기존 24단어급 V6 모델과 117클래스 V7 모델의 모바일 적용 가능성을 비교한다.
- 발표 시연에서 사용할 수 있는 안정적인 gloss 단어와 2~3단어 이상 문장 조합을 찾는다.
- 단어 감지 시점, 최종 문장 확정 시점, 채팅/TTS 출력 시점을 서비스 관점에서 정리한다.

## 기준 파이프라인
- 입력: MediaPipe Holistic landmark
- feature: `30 x 332` RAW landmark sequence
- 모델: TCN 기반 TFLite 모델
- hand forward-fill: 비활성화
- 최소 추론 프레임: 15프레임
- 최대 추론 프레임: 30프레임
- 단어 안정화: 최근 예측 window에서 동일 gloss가 안정적으로 반복될 때 `Detected` 처리
- 문장 확정: 손 미검출 또는 손 비활성 상태가 일정 시간 지속되면 sentence buffer를 최종 문장으로 방출

## 주요 코드/로직 변경

### 1. 최종 문장 확정 전까지 채팅/TTS 지연
- 기존에는 실제 모드에서 단어가 `Detected`되기만 해도 채팅과 음성이 나올 수 있었다.
- 서비스 관점에서는 사용자가 수어 문장을 끝내기 전 중간 단어가 바로 발화되는 것이 부자연스럽다.
- 따라서 `Prediction` 이벤트에서는 채팅/TTS를 실행하지 않고, `Utterance` 이벤트, 즉 최종 문장이 확정된 뒤에만 sLLM 변환과 음성 출력을 수행하도록 변경했다.

### 2. 서버 오디오 지연 대응
- 서버 번역 응답에서 `audioBase64`가 비어 있으면, 기존에는 음성이 나오지 않거나 다음 STT 흐름으로 넘어가는 문제가 있었다.
- 서버가 음성 파일을 주지 않더라도 `correctedText`가 있으면 기기 TTS로 fallback하도록 수정했다.
- 서버 번역 요청은 6초 timeout을 적용했다.
- 목적은 AUTO/서버 모드에서 응답이 1분 가까이 지연되는 현상을 막고, 시연 중 음성 출력 지연을 줄이는 것이다.

### 3. 단어 전환 구간 오염 완화
- 단어가 `Detected`된 뒤에도 이전 단어 프레임이 `SignSequenceBuffer`에 남아 다음 단어 추론에 섞일 수 있었다.
- 안정 단어가 문장 버퍼에 추가되면, 최종 문장 버퍼는 유지하되 라이브 추론용 sequence buffer는 초기화하도록 변경했다.
- 목적은 `기차 -> 놀다 -> 주다`처럼 연속 수어를 할 때 이전 단어의 마지막 자세가 다음 단어 top1에 영향을 주는 현상을 줄이는 것이다.

### 4. no-hands 종료 지연 조정
- no-hands 종료 지연을 너무 짧게 두면 마지막 단어가 안정화되기 전에 최종 문장이 확정될 수 있다.
- 특히 `주다`처럼 손을 내리는 동작은 손 미검출/비활성으로 빨리 판단될 위험이 있다.
- 기본 no-hands 종료 지연은 1.5초로 조정했다.

## V7 초기 모바일 테스트에서 확인한 문제

### 기차/서점 혼동
- `기차` 단독 테스트에서 손 위치가 조금 낮으면 `서점`으로 나오는 경우가 있었다.
- 손 위치가 상대적으로 위로 올라가면 `기차`로 안정적으로 나오는 경향이 있었다.
- V7 모델이 수어의 동작 변화보다 손의 위치/프레임 내 좌표에 크게 반응하는 것으로 보인다.

### 놀다/입원/돼지고기/소개 혼동
- `놀다`는 단독으로는 잘 되는 경우가 많았지만, 연속 문장에서는 `입원`, `돼지고기`, `소개` 등으로 튀는 구간이 있었다.
- 특히 천천히 동작하면 최근 30프레임만 추론에 반영되면서 마지막 자세 위주로 판단되는 느낌이 있었다.

### 주다 누락
- `주다`는 단독 성공 로그도 있었지만, 연속 문장에서는 마지막 단어로 들어갈 때 최종 문장이 먼저 확정되는 문제가 있었다.
- 원인 후보는 두 가지다.
- 첫째, MediaPipe가 오른손을 화면에서 보이는 것과 별개로 landmark를 놓쳐 `right=0`을 반환하는 순간이 있다.
- 둘째, `hasHands=true`여도 한 손이 없고 다른 손이 아래에 있으면 `areBothHandsInactiveByHeight()`에 의해 no-active 상태로 처리될 수 있다.
- 따라서 `주다`는 현재 시연 주력 단어에서 제외하는 편이 안전하다.

### 음성 출력 지연
- 최종 문장 로그가 나온 뒤 음성이 1분 가까이 늦게 나오는 현상이 있었다.
- 원인은 on-device 추론 자체보다 서버 번역/TTS 응답 지연 가능성이 높았다.
- 서버 응답 timeout과 기기 TTS fallback을 적용해 지연을 줄였다.

## 카메라 거리/프레이밍 관찰
- 너무 가까우면 손목이나 손 전체가 프레임 밖으로 잘려 landmark가 끊길 수 있다.
- 너무 멀면 손가락과 손목 landmark가 작아져 안정성이 떨어진다.
- 추천 프레이밍:
  - 얼굴, 양어깨, 가슴-허리 위쪽이 화면에 들어올 것
  - 손을 아래로 내려도 손목이 화면 밖으로 나가지 않을 것
  - 사람 상반신이 화면 세로 높이의 약 60~80%를 차지할 것

## 현재 시연 후보

### 1순위
- `엄마 아프다 치료`
  - 한국어 예시: 엄마가 아파서 치료가 필요해요.
  - 장점: 세 단어 모두 안정적이고 sLLM 변환 효과가 분명하다.
- `시험 모르다 위로`
  - 한국어 예시: 시험을 잘 몰라서 위로가 필요해요.
  - 장점: 세 단어 모두 안정적이고 문장 의미가 자연스럽다.

### 2순위
- `엄마 서점 서다`
  - 한국어 예시: 엄마가 서점에 서 있어요.
  - 장점: 실제 최종 문장으로 나온 조합이다.
  - 단점: 의도한 `독서`가 아니라 `서다`로 바뀐 결과라, 시연 의도에 맞는지는 별도 판단이 필요하다.

### 보류
- `엄마 서점 독서`
- `서점 가다 독서`
- `병원 치료 받다`
- `우유 주다 감사`
- `엄마 병원 가다`

## 현재 결론
- V7 모델은 117클래스 확장 자체는 모바일에서 정상적으로 동작한다.
- 단일 단어 기준으로는 성공 단어가 충분히 있지만, 연속 문장에서는 단어 전환 구간과 마지막 30프레임 중심 추론의 영향이 크다.
- 발표 시연 목표는 전체 단어 커버리지가 아니라 “안정적인 3단어 이상 문장 + sLLM 자연어 변환 + 음성 출력”으로 잡는 것이 현실적이다.
- 현재 가장 안전한 시연 문장은 `엄마 아프다 치료`, `시험 모르다 위로`이다.
- `주다`, `받다`, `감사`, `병원`은 서비스 의미는 좋지만 현재 모델/파이프라인에서는 실패 위험이 높아 시연 주력에서 제외한다.

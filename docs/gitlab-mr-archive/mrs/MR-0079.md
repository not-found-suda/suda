### nej0720 at 2026-05-06T12:21:46.158+09:00 (comment)

```text
**\[P2\] display_text 제약을 엔티티에도 맞춰주세요**

DB 스키마상 learn.display_text는 NOT NULL인데, 엔티티에서는 nullable = false가 빠져 있습니다. 이후 저장 로직이 추가됐을 때 null 값이 객체 상태로는 허용됐다가 flush 시점에 DB 제약 오류로 늦게 터질 수 있어요. word와 동일하게 @Column(nullable = false, length = 100)로 맞춰두면 스키마와 엔티티 제약이 일관돼서 좋을 것 같습니다!
```

### nej0720 at 2026-05-06T12:41:32.070+09:00 (comment)

```text
**\[P1\] learn 스키마와 엔티티 매핑 불일치**

새 `Learn` 엔티티는 `category_id` 기반의 `LearnCategory` 관계로 바뀌었는데, 기존 `learn` 테이블에는 아직 `category` 문자열 컬럼이 `NOT NULL`로 남아있는 것 같습니다. 이 상태면 새 `Learn`을 저장할 때 엔티티에 매핑되지 않은 `category` 컬럼 값이 비어서 INSERT가 실패할 수 있어요.

또 기존 데이터도 `category_id`가 backfill되지 않아서 `findByCategory_Id...` 조회 시 기존 단어들이 결과에서 빠질 가능성이 있어 보입니다. 마이그레이션에서 기존 `category` 값을 `learn_categories`로 이관하고, 그 값을 기준으로 `learn.category_id`를 채운 뒤 제약 조건과 인덱스를 새 구조에 맞게 정리해주면 좋을 것 같습니다!
```

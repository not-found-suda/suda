```text
\[P1\] 원격 GPU AI 서버를 써도 backend가 로컬 ai-server 상태에 묶입니다

docker-compose.ai.yml로 AI 서버를 GPU 서버에 단독 배포하고, Spring은 BACKEND_SIGN_AI_BASE_URL로 원격 AI 서버를 바라보게 하는 구조가 추가됐습니다. 그런데 메인 docker-compose.yml에서는 backend가 여전히 로컬 ai-server의 service_healthy를 필수 의존성으로 갖고 있습니다. Spring 서버에서 원격 GPU AI 서버를 사용하려고 BACKEND_SIGN_AI_BASE_URL=http://\<gpu-private-ip\>:8000을 설정해도, 로컬 ai-server 컨테이너가 모델 artifact 없이 healthcheck에 실패하면 backend 자체가 뜨지 않을 수 있습니다. 원격 AI 서버를 쓰는 배포에서는 로컬 ai-server를 profile/override로 선택적으로 띄우거나, backend depends_on에서 분리할 필요가 있어 보입니다.

\[P2\] Docker Compose backend용 원격 AI URL 환경변수명이 문서와 다릅니다

README의 GPU 서버 단독 배포 섹션에서는 Spring 서버에 SIGN_AI_BASE_URL=http://\<gpu-private-ip\>:8000을 설정하라고 안내합니다. 하지만 Docker Compose로 backend를 실행하는 경우 실제 컨테이너의 SIGN_AI_BASE_URL은 BACKEND_SIGN_AI_BASE_URL 값에서 주입됩니다. Compose 배포자가 문서대로 SIGN_AI_BASE_URL만 설정하면 backend 컨테이너에는 적용되지 않고 기본값인 http://ai-server:8000을 계속 사용할 수 있습니다. Spring 직접 실행과 Docker Compose 실행을 나눠서, Compose에서는 BACKEND_SIGN_AI_BASE_URL을 설정해야 한다고 문서화하는 게 안전해 보입니다.
```

```text
**[P3] 소유권 검증에 사용한 repository 메서드 호출도 확인하면 좋겠습니다**

현재 테스트는 결과적으로 `CHILD_PROFILE_NOT_FOUND`가 발생하는지만 확인합니다. 그런데 이러면 서비스 코드가 실수로 `findByIdAndUserIdAndActiveTrue()`가 아닌 다른 조회 메서드를 호출하도록 바뀌어도, mock 기본값 때문에 테스트가 비슷하게 실패하며 통과할 여지가 있습니다.

소유권 검증 테스트라면 핵심 쿼리인 아래 메서드가 호출됐는지도 검증하는 게 좋겠습니다.

`` `java
verify(childProfileRepository).findByIdAndUserIdAndActiveTrue(CHILD_ID, USER_ID);
`` `

특히 이 테스트의 목적이 “다른 보호자의 아이 또는 비활성 아이를 막는다”라면, `userId`와 `activeTrue` 조건이 들어간 조회를 사용한다는 점을 테스트에 명확히 고정하는 편이 더 좋습니다.
```

### hh001204 at 2026-05-12T19:37:50.440+09:00 (comment)

```text
**[P3] 소유권 검증 테스트 대상이 일부 메서드에만 한정되어 있습니다**

이번 테스트는 getChild, deleteChild, createSession, getCurrentQuestion, getSummary 정도만 소유권 거부 케이스를 확인하고 있습니다. 그런데 실제로 childId나 sessionId 기반으로 소유권을 타야 하는 메서드는 더 있습니다.

예를 들면:
`` `java
ChildProfileService.updateChild()
QuizService.submitAnswer()
QuizService.getResult()
QuizService.endSession()
ReportService.getWeakWords()
ReportService.getCategories()
ReportService.getQuizSessions()
ReportService.getQuizSessionDetail()
`` `

브랜치 목적이 “아이 소유권 검증 테스트 추가”라면, 최소한 퀴즈 결과/종료와 리포트 목록/상세 쪽도 한두 개 더 잡아두는 편이 더 안전해 보입니다.
```

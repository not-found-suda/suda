## Description

## 요약

* 네이버 OAuth 로그인을 authorization code + PKCE 기반으로 구현했습니다.

## 작업 배경

- 관련 이슈:  S14P31A404-313
- 참고 문서/기획:

## 주요 변경 사항

* POST /api/v1/auth/oauth/naver API를 추가했습니다.
* 네이버 authorization code를 access token으로 교환한 뒤 프로필을 조회하도록 구현했습니다.
* 최초 소셜 로그인 시 users, social_accounts를 생성하고, 기존 소셜 계정은 로그인 처리합니다.
* 로그인 성공 응답은 일반 로그인과 동일하게 accessToken, refreshToken, tokenType, expiresIn을 반환합니다.


- 변경하지 않은 것 / 스코프 아웃
  * 카카오 OAuth 로그인은 이번 작업 범위에서 제외했습니다.
  * 소셜 계정 목록 조회/연동 해제 API는 별도 이슈에서 처리 예정입니다.

## 스크린샷/로그

* API는 모바일에서 실제 네이버 인증 후 받은 authorizationCode, state, codeVerifier가 있어야 성공 응답 캡처가 가능합니다.\
  현재 백엔드 단독으로는 .env 키 로딩 후 테스트 통과까지 확인했습니다

### soshyzx123 at 2026-05-08T14:09:04.686+09:00 (comment)

```text
전체적인 네이버 OAuth 로그인 방향은 좋아 보입니다. :grin: 

일반 로그인과 동일한 accessToken/refreshToken 응답을 재사용하는 점도 모바일 연동 관점에서 자연스럽습니다.

다만 모바일 연동 전에 요청 Body 계약은 한 번 맞춰야 할 것 같습니다. 공유받은 모바일 전달값은 authorizationCode, codeVerifier, redirectUri인데, 현재 DTO는 redirectUri 없이 state를 필수로 받고 있습니다. 최종 계약이 state 기반이면 모바일 명세를 state로 수정해야 하고, redirectUri 기반이면 DTO와 Naver token exchange form에 redirectUri 반영이 필요해 보입니다.

추가로 네이버 API 호출에서 타임아웃/연결 실패 같은 RestClientException이 OAUTH_PROVIDER_ERROR로 매핑되도록 보강하면 운영 중 외부 Provider 장애 대응이 더 안정적일 것 같습니다.

Swagger 설명도 현재 "Access Token" 방식으로 적혀 있어 authorization code + PKCE 설명으로 바꿔주시면, 모바일 구현자가 헷갈리지 않을 것 같습니다! :grin: :grin:

**P2: 네이버 API 네트워크 오류가 500으로 떨어질 수 있습니다**

현재 코드는 `RestClientResponseException`만 잡고 있어서 타임아웃/DNS/연결 실패는 500으로 떨어질 수 있습니다. `RestClientException` 또는 `ResourceAccessException`까지 잡아 `OAUTH_PROVIDER_ERROR`로 매핑하면 더 안전합니다.
```

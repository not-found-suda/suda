```text
수어 인식의 핵심인 시퀀스 버퍼와 정규화 인코더 구현하시느라 고생 많으셨습니다!

특히 ArrayDeque를 이용한 슬라이딩 윈도우 버퍼와, 손이 가려졌을 때 이전 값을 사용하는 보간 로직(fillLandmarks)은 수어 인식의 정확도를 높여줄 아주 좋은 선택인 것 같습니다. 👍👍

다만, 성능 최적화 관점에서 두 가지만 의견 드려봅니다! :goat: 

p5. 

**객체 재사용**: LandmarkFeatureEncoder의 encode 함수가 매 프레임마다 FloatArray와 LandmarkNormalizer 객체를 새로 생성하고 있는데, 카메라 프레임 수(30FPS)를 고려하면 가비지 컬렉션(GC) 부하가 발생할 우려가 있어 보입니다. 가능하면 values 배열을 필드로 두고 값을 덮어쓰는 구조로 개선해 보면 어떨까요?

**Thread-safety**: SignSequenceBuffer는 카메라 분석 스레드에서 비동기적으로 접근될 텐데, ArrayDeque에 대한 스레드 안전성 처리가 추가되면 더 견고한 엔진이 될 것 같습니다!

코드 설계가 워낙 탄탄해서 이 부분들만 개선되면 완벽할 것 같네요. Approve 합니다! 😊😊
```

```text
좋은 포인트 감사합니다! 👁️👁️

현재는 프레임별 feature 값이 sequence buffer에 안전하게 저장되는 것을 우선해서 encode마다 새 FloatArray를 반환하도록 구현했습니다.👃

배열을 바로 재사용하면 이전 프레임 값이 덮어써질 수 있어서, MediaPipe/TFLite 연결 후 실제 FPS/latency를 확인하면서 필요할 때 최적화하겠습니다.👄

---

Thread-safety 부분도 좋은 지적 감사합니다! 👁️👁️

현재는 ImageAnalysis analyzer가 singleThreadExecutor 기반이라 프레임 처리는 순차적으로 실행됩니다.👃

다만 추후 MediaPipe/TFLite 연결로 처리 스레드가 나뉘면 stop/reset과 frame 처리 타이밍이 겹칠 수 있으므로, 이번 MR이 아니라 다음 MR 또는 실제 추론 파이프라인 연결 시점에 buffer lock이나 단일 dispatcher 방식으로 보강하겠습니다.👄
```

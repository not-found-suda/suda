# 커밋 메시지 컨벤션 (Commit Convention)

본 프로젝트는 원활한 협업과 AI 에이전트의 정확한 컨텍스트 파악을 위해 아래의 커밋 메시지 규칙을 엄격하게 준수합니다.
*(지라 이슈 트래킹은 MR(Merge Request) 본문에서 진행하므로, 커밋 메시지에는 이슈 번호를 포함하지 않습니다.)*

## 1. 기본 구조
커밋 메시지는 제목(Title), 본문(Body), 꼬리말(Footer)로 구성하며, 각 항목은 빈 줄로 구분합니다.

```text
type: 제목 (50자 이내)
<빈 줄>
본문 (선택 사항, 상세한 변경 이유와 내용 설명)
<빈 줄>
꼬리말 (선택 사항, 기타 참고 사항)
```

## 2. 커밋 타입 (Type)
반드시 아래에 지정된 타입 중 하나를 소문자로 사용합니다.

| 타입 | 설명 | 예시 |
| :--- | :--- | :--- |
| `feat` | 새로운 기능 추가 | `feat: 카카오 소셜 로그인 기능 추가` |
| `fix` | 버그 및 오류 수정 | `fix: STT 인식 결과 지연 현상 수정` |
| `docs` | 문서 작성 및 수정 (README, 기획 등) | `docs: 커밋 컨벤션 및 깃플로우 가이드 추가` |
| `style` | 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음) | `style: ktlint 코드 포맷팅 적용` |
| `refactor` | 코드 리팩토링 (기능 변화 없음, 구조 개선) | `refactor: 영상 프레임 추출 로직 분리` |
| `test` | 테스트 코드 추가 및 수정 | `test: 수어 인식 모델 성능 테스트 추가` |
| `chore` | 빌드 설정, 패키지 매니저 관리, CI/CD 설정 | `chore: build.gradle 의존성 업데이트` |

## 3. 세부 작성 규칙 (Rule)

### 제목 (Title)
- **이슈 번호 생략:** 제목에 지라 티켓 번호를 포함하지 않습니다. (MR 생성 시 연동)
- **길이 제한:** 50자 이내로 간결하게 작성합니다.
- **마침표 금지:** 제목 끝에 마침표(`.`)를 찍지 않습니다.
- **명사형/명령조 사용:** "~추가", "~수정", "~적용" 등의 명사형 어미로 끝냅니다. (예: `추가함`(X) -> `추가`(O))

### 본문 (Body)
- **선택 사항:** 제목만으로 설명이 부족할 때 작성합니다.
- **'어떻게'보다 '왜'에 집중:** 코드가 어떻게 변경되었는지보다, **왜(Why)** 이 변경이 필요했는지, 어떤 문제를 해결했는지 설명합니다. (AI 에이전트가 코드를 리뷰할 때 가장 중요한 지표가 됩니다.)
- **길이 제한:** 한 줄당 72자를 넘지 않도록 줄바꿈을 합니다.

### 꼬리말 (Footer)
- **선택 사항:** 부가적인 참고 링크나, 함께 리뷰해야 할 팀원 멘션 등이 필요할 때만 가볍게 사용합니다.

## 4. 예시 (Example)

```text
feat: 카메라 프레임 분석을 위한 TFLite 모델 연동

- 실시간 영상 프레임 추출을 위해 CameraX 의존성 추가
- TFLite C++ API 대신 Kotlin 바인딩을 활용하여 지연 시간 단축 시도
- NPU 할당 로직은 제외함 (추후 별도 티켓으로 분리 예정)
```

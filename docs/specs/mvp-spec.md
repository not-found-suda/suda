## 1. MVP 핵심 기획
* **목표:** 복잡한 기능 없이, 앱을 켜면 즉시 수어/음성을 인식하고 실시간 자막으로 보여주는 직관적인 대면 소통에 집중.
* **제외된 기능:** 로그인/회원가입, 학습 기록 저장 및 조회 API
* **현 상황:** 온디바이스 AI 도입으로 서버의 트래픽 및 연산 부담이 극적으로 감소. 안드로이드(Kotlin) 앱 내에서의 실시간 미디어 제어 및 AI 모델(TFLite) 구동으로 프로젝트의 핵심 개발 역량이 이동.

## 2. MVP 개발 원칙
1. **클라이언트(App) 중심 성능 최적화:** 서버 의존도를 최소화하고, 카메라 프레임 분석과 1차 텍스트 변환을 기기 내부(NPU/CPU)에서 처리하여 지연 시간(Low Latency)을 극복.
2. **백엔드 역할 축소 (API Gateway):** 무거운 실시간 스트리밍 인프라(Kafka, WebSocket)를 폐기. 백엔드는 LLM(문맥 교정)과 TTS(음성 합성)를 호출하여 결과만 앱으로 반환하는 가벼운 REST API 서버로 동작.
3. **디자인(UI) 최소화:** 앱 UI는 꾸미지 않고 작동(웹캠 렌더링, 텍스트 자막 출력)에만 집중. 안드로이드 기본 Material Design 컴포넌트 활용.

## 4. 🛠 세부 개발 태스크 및 역할 분배

### 4.1. AI 파트 (2명) : [모델 경량화 및 모바일 이식]
> 기존 서버 서빙 방식에서 모바일 탑재 방식으로 목표 전면 수정.
* **[AI] 모델 변환 (TFLite):** 파이썬에서 학습된 LSTM 모델을 안드로이드 기기에서 돌아갈 수 있도록 TensorFlow Lite (.tflite) 포맷으로 변환.
* **[AI] 양자화 및 최적화:** 모바일 환경(NPU/CPU)에 맞춰 연산 속도를 높이고 용량을 줄이기 위해 가중치 양자화(Quantization) 수행. 앱 개발자와 협업하여 프레임당 추론 속도 테스트.
* **담당:** 김순우, 설현원

### 4.2. Android 파트 A (1명) : [카메라 제어 & AI 브릿지]
> 영상 프레임을 추출해 온디바이스 모델에 밀어 넣는 파이프라인의 핵심.
* **[APP] 영상 스트림 제어:** `CameraX` 라이브러리를 활용해 전면 카메라 영상 렌더링 및 초당 프레임(FPS) 조절 로직 구현.
* **[APP] On-Device AI 연동:** C++ 엔진을 거치지 않고 Kotlin 코드 레벨에서 MediaPipe(관절 추출) 및 TFLite(수어 분류) 모델을 직접 호출하여 실시간 수어->단어(Text) 변환 파이프라인 구축.
* **담당:** 김휘수

### 4.3. Android 파트 B (1명) : [음성 인식 & UI/UX 코어]
> 대면 소통 시나리오를 위한 화면 구성과 상대방의 음성 처리 전담.
* **[APP] 로컬 STT 연동:** 자녀의 음성 인식을 위해 안드로이드 내장 `SpeechRecognizer` 또는 경량 구글 STT API를 호출하여 즉시 텍스트로 변환.
* **[APP] 비동기 UI 업데이트:** AI 파트 A에서 넘어온 텍스트와, 서버에서 받아온 교정된 텍스트를 메인 쓰레드 끊김 없이 화면 자막으로 부드럽게 렌더링. 접근성(글자 크기 등) 설정 구현.
* **담당:** 손홍헌 (기존 FastAPI 담당에서 전환)

### 4.4. Android 파트 C (1명) : [네트워크 & 미디어 재생]
> 서버와의 가벼운 통신 및 전달받은 미디어(오디오) 재생 전담.
* **[APP] REST API 통신:** Retrofit2나 Ktor를 활용해, 기기에서 추출된 원본 텍스트를 Spring Boot 서버로 `POST` 전송.
* **[APP] 오디오 파일 재생:** 서버에서 반환된 TTS 오디오 파일(Byte Array 또는 URL)을 안드로이드 `MediaPlayer`를 활용해 딜레이 없이 재생 및 메모리 해제.
* **담당:** 김민선

### 4.5. Spring Boot (1명) : [외부 API 게이트웨이]
> 복잡한 소켓 통신을 걷어내고, 무거운 외부 AI 모델을 중계하는 역할만 수행.
* **[BE] LLM 문맥 교정 API:** 앱에서 올라온 단어 배열(예: `["엄마", "해보다"]`)을 받아, Gemini API를 호출하여 자연스러운 문장("엄마 해봐!")으로 다듬어 반환.
* **[BE] TTS 합성 API:** 교정된 문장을 Clova TTS API로 보내 고품질 음성 데이터를 생성한 뒤, 앱에서 바로 재생할 수 있는 포맷으로 응답.
* **[BE] 발음 보정 파이프라인:** 자녀의 부정확한 STT 텍스트("옴마")를 수신하면, LLM을 통해 유아 발음을 보정("엄마!")하여 응답하는 로직 구현.
* **담당:** 나예지

## 5. MVP 작동 시나리오

**[상황: 농인 부모님과 청인 자녀가 함께 있으며, 단일 태블릿/스마트폰 기기 화면을 부모님이 보고 있음]**

1. **앱 실행 (네트워크 대기 없음)**
   * 앱을 켜면 서버와 소켓 연결을 할 필요 없이, 기기 메모리에 TFLite 모델이 로드되며 즉시 카메라가 켜집니다. 인터넷이 다소 불안정해도 핵심 변환 로직은 즉시 작동할 준비를 마칩니다.
2. **농인 부모님의 발화 (On-Device 수어 → 텍스트)**
   * 부모님이 수어로 "엄마", "해보다"라고 표현합니다.
   * **[On-Device]:** 앱 내부의 CameraX가 영상을 캡처하고, 기기 내장 TFLite 모델이 즉시 수어 단어 `["엄마", "해보다"]`를 인식해 텍스트로 뽑아냅니다. (서버 전송 없음, 트래픽 0)
   * **[API 호출]:** 추출된 텍스트만 Spring Boot 서버로 보냅니다.
   * **[서버 처리]:** Spring은 Gemini를 통해 "엄마 해봐!"로 문맥 교정하고, Clova TTS를 통해 음성 파일을 만들어 앱으로 돌려줍니다.
   * 앱 화면에 "엄마 해봐!" 자막이 뜨고, 스피커로 음성이 출력됩니다.
3. **청인 자녀의 대답 (로컬 음성 → 텍스트)**
   * 자녀가 "옴마!" 라고 대답합니다.
   * **[On-Device]:** 기기의 마이크가 이를 듣고 안드로이드 로컬 STT가 "옴마"라는 텍스트를 추출합니다.
   * **[API 호출 & 서버 처리]:** Spring 서버로 텍스트를 보내면, 발음 보정 로직을 거쳐 "엄마!"라는 정확한 단어로 응답합니다.
   * 부모님이 보는 화면에 즉시 "엄마!" 라는 텍스트 자막이 나타납니다.
4. **단기 기억**
   * 앱을 끄면 기기 메모리에 있던 대화 텍스트는 즉시 휘발되며 저장되지 않습니다. (MVP는 실시간 구현에 집중)

## 🚨 [부록] 온디바이스 전환 시 MVP 기획 수정 로드맵
*(작성일: 2026/04/23 - 코멘트 정리)*

1. **아키텍처 다이어트: 무거운 통신 인프라 전면 폐기**
   * **삭제 대상:** FastAPI (프레임 스트리밍), Kafka (메시지 브로커), K8s 고가용성 인프라.
   * **이유:** 온디바이스는 폰의 카메라에서 찍힌 영상을 폰 안에서 바로 AI가 분석. 1초에 수십 장씩 서버로 이미지를 쏠 필요가 없으므로 대규모 I/O 처리를 위한 파이프라인 자체가 필요 없어집니다.
   * **변경점:** 서버망의 역할이 '실시간 스트리밍 처리'에서 **'가벼운 API 게이트웨이(LLM, TTS 외부 호출용)'**로 축소됩니다.
2. **소통 방식의 재정의: WebSocket(STOMP) 무용론**
   * **현재 기획의 모순:** 시나리오를 보면 "단일 디바이스(태블릿)의 카메라/화면이 부모님을 향해 있음"이라고 되어 있습니다. 기기 1대를 부모와 자녀가 같이 본다면, 데이터를 다른 기기로 쏴줄 필요가 없으므로 **WebSocket 브로드캐스팅 자체가 필요 없습니다.**
   * **변경점:** 실시간 양방향 소켓 통신을 빼고, 기기 내부에서 결과를 바로 화면에 띄우는 구조로 변경합니다. (추후 서로 다른 기기 2대로 대화하는 기능이 추가될 때 소켓을 도입하는 것이 맞습니다.)
3. **인원 및 역할 재배치 (가장 중요한 부분)**
   * 온디바이스 앱은 **안드로이드(프론트엔드) 개발자의 비중이 절대적으로 높아야 합니다.**
   * **AI (2명 유지):** 모델 학습 방식을 모바일용 경량화(TFLite 변환, 양자화)로 목표 변경.
   * **Android/Kotlin (기존 FastAPI/Infra 인력 전환, 2~3명):** CameraX로 영상 제어, TFLite 모델 기기 이식, UI/UX 및 기기 자체 STT(SpeechRecognizer) 구현.
   * **Spring Boot (1~2명으로 축소):** 앱에서 보내는 '텍스트'를 받아 Gemini API로 교정해 주거나, Clova TTS 음성 파일을 만들어 앱으로 돌려주는 **가벼운 REST API 서버** 역할만 수행.
4. **시나리오 데이터 플로우 변경**
   * 서버(Spring/FastAPI) 중심에서 **기기(Device) 중심**으로 서술 방향을 바꿉니다.
   * *기존:* 기기 촬영 → 서버로 프레임 전송 → 서버 내 AI 분석 → 카프카 → 소켓 → 기기 수신
   * *변경:* 기기 촬영 → **기기 내부 AI(TFLite) 즉시 분석 및 텍스트화** → 텍스트만 Spring 서버로 REST API 전송 → 서버에서 Gemini 교정 및 TTS 생성 → 기기 수신 및 재생
